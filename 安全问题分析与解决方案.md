## 二、详细分析与解决状态

### 2.1 跨站脚本（XSS）问题分析

#### 扫描发现的6个XSS问题

| 序号 | 文件位置 | 行号 | 扫描描述 |
|-----|---------|------|---------|
| 1 | ApiSuccess.java | 25, 38, 39 | 不可信数据从setData()方法返回 |
| 2 | AuthRequests.java | 24, 70, 149 | username字段返回不可信数据 |
| 3 | BaseAuthResponse.java | 11, 14, 16 | tenantId/userId/username返回不可信数据 |
| 4 | BaseTenantRequest.java | 23 | tenantId字段返回不可信数据 |
| 5 | TenantRequests.java | 13 | tenantId/name字段返回不可信数据 |
| 6 | TenantResponse.java | 11 | tenantId字段返回不可信数据 |

#### ✅ 分析结论：**误报（False Positive）**

**原因分析：**

1. **系统架构特征**
   - ✅ 纯粹的 RESTful JSON API，使用 `@RestController`
   - ✅ 不存在任何 HTML/JSP/Thymeleaf 视图渲染
   - ✅ 所有响应通过 Jackson 序列化为 JSON
   - ✅ Content-Type 固定为 `application/json`

2. **XSS攻击前提条件不满足**
   - ❌ 无 HTML 上下文：数据不会插入到 HTML 标签或属性中
   - ❌ 无模板引擎：不存在服务器端模板注入风险
   - ❌ 无脚本执行：JSON 数据不会被浏览器当作代码执行

3. **现有安全措施**
   - ✅ 输入验证：所有字段都有严格的 `@Pattern` 正则验证
   - ✅ 租户ID格式：仅允许 `[a-zA-Z0-9_-]+`
   - ✅ 用户名格式：严格的字母数字组合
   - ✅ JSON序列化：自动转义特殊字符

**技术说明：**

```java
// 示例：DTO 结构（JSON API）
@RestController  // 不是 @Controller，不会渲染 HTML
public class AuthController {
    @PostMapping("/api/auth/login")
    public ResponseEntity<ApiResponse<AuthResponse>> login(@RequestBody LoginRequest req) {
        // 返回的数据会被 Jackson 自动序列化为 JSON
        return ResponseEntity.ok(ApiResponse.success(response));
    }
}

// 客户端收到的是纯 JSON（不会执行）：
{
  "success": true,
  "data": {
    "token": "eyJhbGc...",
    "username": "user123"  // 即使包含<script>也不会执行
  }
}
```

**对比真实XSS场景：**

```java
// ❌ 真正的XSS风险场景（本项目不存在）
@Controller  // 注意：不是 RestController
public class UnsafeController {
    @GetMapping("/user")
    public String getUserPage(Model model, @RequestParam String name) {
        model.addAttribute("username", name);  // 未编码的数据
        return "user-profile";  // 渲染 HTML 模板
    }
}

<!-- user-profile.html -->
<h1>Welcome, <span th:text="${username}"></span></h1>
<!-- 如果 username = "<script>alert(1)</script>" 则会执行 -->
```

#### ✅ 解决方案：**无需修复（误报）**

**建议操作：**
1. 在安全扫描工具中将这些 DTO 类添加到白名单
2. 配置扫描工具识别 REST API 架构（不检查纯 JSON 响应）
3. 文档化说明：Mercury-Auth 是纯 JSON API，不存在 HTML 渲染

#### 2.2.2 注释中的密码（1个）

**检查结果：**

扫描工具可能识别到 SQL schema 文件中的字段注释：

```sql
-- schema.sql
CREATE TABLE users (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(50) NOT NULL,
  password_hash VARCHAR(200) NOT NULL,  -- ⚠️ 扫描工具可能误报此注释
  ...
);
```

**分析：**

这是 **误报**。注释说明的是 "password_hash" 字段，表示密码哈希值，不是实际密码。

#### ✅ 结论：**误报（注释仅说明字段用途）**

**解决方案：**
- 无需修复（正常的数据库schema注释）
- SQL文件中不包含任何实际密码值
- 所有密码都使用BCrypt哈希存储
