# SSO 可用性分析

## 当前框架能做什么
- 统一的登录入口，支持用户名/密码、邮箱验证码、短信验证码、微信登录（目前为 openId 直传存根）。
- 签发 JWT 并对访问进行校验，提供刷新、黑名单、审计日志、租户隔离与限流等能力。

在**所有业务系统都直接调用本服务、并统一使用 JWT 进行鉴权**的前提下，可以实现“轻量级的统一登录体验”（类似集中登录 + token 分发），但仍不等同于主流 SSO。

## 场景分析：多站点租户组（A/B 互通，C/D 独立）
- **现状**：系统要求 `X-Tenant-Id` 与 JWT 中的 tenantId 严格一致，默认不支持跨租户 token 互通。
- **结论**：若 A/B 需要互通，**当前模块无法仅靠“配置一张表”直接实现**；除非把 A/B 放在同一个 tenantId（同租户即共享用户池），否则 token 不会被 B 接受。
- **可行路径（需要开发）**：
  1. 增加“租户组”表（tenant_group / tenant_group_member），定义 A/B 为同组。
  2. 在 JWT 校验处允许 **token 的 tenantId 属于同组**，并在授权层区分具体站点权限。
  3. 统一用户标识与账号生命周期（例如同组内共享 username/email/phone 的唯一性）。
  4. 处理跨站点登出、黑名单与限流的组维度策略。

## 与主流统一认证/SSO 的差距
1. **缺少标准协议**：没有 OAuth2/OIDC 授权码流程、token endpoint、JWKS、scope/consent；也不支持 SAML 2.0/CAS 等企业 SSO 协议。
2. **缺少客户端/应用注册与跳转式登录**：主流 SSO 需要客户端注册、redirect_uri 校验、授权码交换等能力。
3. **缺少浏览器会话与单点登出**：没有统一会话 cookie、前/后端通道 Single Logout、会话生命周期管理。
4. **联邦身份与第三方 IdP 对接不足**：微信 OAuth2 仅存根，缺少企业 IdP（如 Azure AD、Keycloak、Okta）对接。
5. **权限/授权模型不完整**：缺少标准化 scope、RBAC/ABAC 授权中心能力与资源服务器授权策略。
6. **用户/账号生命周期管理不足**：缺少 SCIM/自动化用户同步、账号绑定、审计合规报表等企业功能。

## 结论与建议
- **结论**：当前框架更适合作为“集中登录 + JWT 发行服务”，可覆盖自有系统的统一登录；若要作为**标准 SSO/统一认证中心**，仍需补齐协议与联邦集成能力。
- **建议路径**：优先补齐 OAuth2/OIDC（可参考 Spring Authorization Server），再考虑 SAML 2.0、Single Logout 与第三方 IdP 对接；同时完善客户端注册、授权范围与用户生命周期管理。
